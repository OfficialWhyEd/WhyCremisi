Paper 12 — Deployment & Distribution
CI/CD, Code Signing, Notarization and Installer

ABSTRACT: This paper describes the complete distribution pipeline for WhyCremisi,
from build automation to final publication. It covers semantic versioning,
multi-platform compilation, code signing, macOS notarization, installer creation,
and GitHub Actions automation. The goal is to ensure a secure, repeatable, and
verifiable release process on macOS, Windows, and Linux.


1 — RELEASE CYCLE OVERVIEW

WhyCremisi adopts Semantic Versioning 2.0.0 (MAJOR.MINOR.PATCH):

  Component | When to increment                   | Example
  MAJOR     | Breaking API/plugin change           | 2.0.0
  MINOR     | New feature, backward-compatible     | 1.3.0
  PATCH     | Bug fix, no new features             | 1.2.1

The release cycle follows four progressive stages:
  Nightly -> Beta -> RC -> Stable

hotfix/* branches branch off main and are merged back into both main and develop
to prevent regressions.


2 — BUILD AUTOMATION

WhyCremisi uses CMake presets to standardize builds across all systems.

Presets: debug, release, relwithdebinfo

On macOS: CMake with Xcode generator, CMAKE_OSX_ARCHITECTURES = "x86_64;arm64"
to produce a universal binary. Code signing with Developer ID Application cert.

On Windows: CMake with "Visual Studio 17 2022" generator, signing with signtool.exe
and RFC 3161 timestamping.

On Linux: CMake with Ninja generator, packaging with CPack (DEB and RPM).


3 — MACOS CODE SIGNING

Code signing is mandatory for distributing audio plugins on macOS.

Certificate: Apple Developer ID Application (3-year validity).
Entitlements: disable-library-validation, allow-unsigned-executable-memory,
allow-dyld-environment-variables.

Hardened Runtime enabled via --options runtime.


4 — WINDOWS CODE SIGNING

Extended Validation (EV) Code Signing certificate.

Signtool:
  signtool sign /fd SHA256 /tr http://timestamp.digicert.com /td SHA256 /a /v

Windows Hardware Compatibility Program (WHCP) can provide additional Microsoft
signature for audio drivers.


5 — MACOS NOTARIZATION

Notarization is Apple's process for verifying software is free of malicious code.

Pipeline:
  1. ditto -c -k to create zip
  2. xcrun notarytool submit with apple-id, team-id, password
  3. xcrun notarytool log to check status
  4. xcrun stapler staple to attach ticket to DMG


6 — INSTALLER

macOS: DMG with custom background, Applications symlink, EULA,
component plist (VST3/AU/Standalone).

Windows: Inno Setup with selectable components: VST3 64-bit, AAX, Standalone.

Linux: .deb and .rpm packages generated with CPack.


7 — DISTRIBUTION CHANNELS

Direct downloads from website with SHA-256 checksums and GPG signature.

Auto-update:
  - macOS: Sparkle (appcast XML + signed DMG)
  - Windows: Squirrel (.nupkg + Release EXE)

License key validation: POST /validate, JWT token response (30-day validity).
Telemetry opt-in via Crashpad.


8 — RELEASE AUTOMATION

GitHub Actions workflow with matrix build on 3 OS:
  1. Checkout + setup Python
  2. CMake configure + build with release preset
  3. Code signing (macOS: cert.p12 + keychain; Windows: cert.pfx + signtool)
  4. Notarization (macOS: notarytool + stapler)
  5. Packaging (CPack)
  6. Upload to GitHub Release

Changelog generated by git-cliff based on conventional commits.

Draft release PR toward main triggers staging build, test suite, and release
notes draft.


9 — POST-RELEASE VERIFICATION

Smoke test:
  - codesign --verify and spctl on macOS
  - signtool verify on Windows
  - sha256sum verify checksum

Malware scan via VirusTotal API, XProtect, Windows Defender.

DAW compatibility smoke test on: Ableton Live 12, Logic Pro 11, Pro Tools 2024,
FL Studio 21, Cubase 13, REAPER 7, Studio One 6.
